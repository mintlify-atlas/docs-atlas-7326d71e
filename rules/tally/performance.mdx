tally performance rules suggest BuildKit features that reduce image size, speed up builds, and improve layer caching. Most are auto-fixable with `--fix --fix-unsafe`.

## tally/prefer-add-unpack 🔧

| | |
|---|---|
| **Severity** | Info |
| **Default** | Enabled |
| **Auto-fix** | Yes (unsafe — `--fix --fix-unsafe`) |

Suggests `ADD --unpack <url> <dest>` instead of `RUN curl/wget + tar` patterns for downloading and extracting remote archives. BuildKit implements `ADD --unpack` natively in Go, reducing image size and eliminating the need to spawn download and extraction processes inside the build container.

Works on Windows containers too, and supports PowerShell `Invoke-WebRequest` patterns.

**Detected patterns:**

- Pipe: `curl -fsSL <url> | tar -xz -C /dest`
- Download-then-extract: `curl -o /tmp/app.tar.gz <url> && tar -xf /tmp/app.tar.gz -C /dest`
- `wget` variants
- Windows `curl.exe` and PowerShell `Invoke-WebRequest` variants

**Violation:**

```dockerfile
FROM ubuntu:22.04
RUN curl -fsSL https://go.dev/dl/go1.22.0.linux-amd64.tar.gz | tar -xz -C /usr/local

RUN wget -O /tmp/node.tar.xz https://nodejs.org/dist/v20.11.0/node-v20.11.0-linux-x64.tar.xz \
    && tar -xJf /tmp/node.tar.xz -C /usr/local --strip-components=1
```

**Fix:**

```dockerfile
FROM ubuntu:22.04
ADD --unpack https://go.dev/dl/go1.22.0.linux-amd64.tar.gz /usr/local

ADD --unpack https://nodejs.org/dist/v20.11.0/node-v20.11.0-linux-x64.tar.xz /usr/local
```

<Note>
The auto-fix is only emitted when the `RUN` instruction contains **only** the download and extraction commands. If additional commands are present (e.g., `chmod`, `mv`), the violation is reported but no fix is suggested.
</Note>

## tally/prefer-copy-heredoc 🔧

| | |
|---|---|
| **Severity** | Info |
| **Default** | Enabled |
| **Auto-fix** | Yes (unsafe — `--fix --fix-unsafe`) |

Suggests replacing `RUN echo/cat/printf > file` patterns with `COPY <<EOF` heredoc syntax. `COPY` does not spawn a shell container, making it faster. `COPY --chmod` also sets permissions atomically in a single layer.

**Detected patterns:**

- `echo "content" > /path/to/file`
- `printf 'line1\nline2\n' > /path/to/file`
- `echo "x" > /file && chmod 0755 /file`
- BuildKit heredoc piped to `cat` or `tee`
- Consecutive `RUN` instructions writing to the same file

**Violation:**

```dockerfile
RUN printf '#!/bin/sh\nexec nginx -g "daemon off;"\n' > /usr/local/bin/start-nginx \
    && chmod 0755 /usr/local/bin/start-nginx

RUN apt-get update \
    && echo "APP_ENV=production" > /etc/myapp.env \
    && apt-get clean
```

**Fix:**

```dockerfile
COPY --chmod=0755 <<EOF /usr/local/bin/start-nginx
#!/bin/sh
exec nginx -g "daemon off;"
EOF

RUN apt-get update
COPY <<EOF /etc/myapp.env
APP_ENV=production
EOF
RUN apt-get clean
```

**Configuration:**

```toml
[rules.tally.prefer-copy-heredoc]
check-single-run = true
check-consecutive-runs = true
```

## tally/prefer-multi-stage-build 🔧

| | |
|---|---|
| **Severity** | Info |
| **Default** | Off (experimental) |
| **Auto-fix** | Yes (AI-powered unsafe fix via ACP) |

Suggests converting single-stage builds into multi-stage builds to separate build tooling from the runtime image, reducing the final image size.

The rule triggers when the Dockerfile has exactly one `FROM` stage and contains at least one of:

- A build-related package install (e.g., `apt-get install build-essential`, `apk add gcc`)
- A build step (e.g., `go build`, `cargo build`, `npm run build`, `dotnet publish`)
- A download-and-install pattern (e.g., `curl ... | tar ...`)

<Note>
The auto-fix for this rule requires `--fix --fix-unsafe` **and** a configured ACP-capable AI agent. The fix rewrites the entire Dockerfile, so review carefully before applying.
</Note>

**Configuration:**

```toml
[rules.tally.prefer-multi-stage-build]
severity = "info"
min-score = 4
fix = "explicit"
```

Setting `fix = "explicit"` is recommended to prevent accidental AI rewrites when running `--fix-unsafe` broadly.

## tally/prefer-package-cache-mounts 🔧

| | |
|---|---|
| **Severity** | Info |
| **Default** | Off (experimental) |
| **Auto-fix** | Yes (unsafe — `--fix --fix-unsafe`) |

Suggests BuildKit `--mount=type=cache` mounts for package manager install and build commands. Cache mounts persist the package manager's download cache across builds, dramatically speeding up rebuilds without bloating image layers.

The rule also removes cache-cleaning commands and cache-disabling flags that cancel out the benefit of cache mounts (e.g., `apt-get clean`, `--no-cache-dir`, `pip cache purge`).

**Supported package managers:** apt/apt-get, apk, dnf, yum, zypper, npm, yarn, pnpm, pip, bundle (gem), cargo, dotnet (nuget), composer, uv, bun.

**Violation:**

```dockerfile
FROM ubuntu:24.04
RUN apt-get update && apt-get install -y gcc && apt-get clean
```

**Fix:**

```dockerfile
FROM ubuntu:24.04
RUN --mount=type=cache,target=/var/cache/apt,id=apt,sharing=locked \
    --mount=type=cache,target=/var/lib/apt,id=aptlib,sharing=locked \
    apt-get update && apt-get install -y gcc
```

**pnpm with `PNPM_HOME`:**

```dockerfile
FROM node:20-slim
ENV PNPM_HOME="/pnpm"
# Violation: no cache mount
RUN pnpm install --frozen-lockfile && pnpm store prune
```

```dockerfile
FROM node:20-slim
ENV PNPM_HOME="/pnpm"
# Fixed: cache mount added, pnpm store prune removed
RUN --mount=type=cache,target=/pnpm/store,id=pnpm \
    pnpm install --frozen-lockfile
```

**Enable in `.tally.toml`:**

```toml
[rules.tally.prefer-package-cache-mounts]
severity = "info"
```

## tally/php/composer-no-dev-in-production 🔧

| | |
|---|---|
| **Severity** | Warning |
| **Default** | Enabled |
| **Auto-fix** | Yes (suggestion — `--fix --fix-unsafe`) |

Production Composer install commands should include `--no-dev` to exclude development dependencies. Shipping `require-dev` packages increases the dependency graph, image size, and attack surface.

Stages explicitly named `dev`, `development`, `test`, `testing`, `ci`, or `debug` are skipped. The rule also accepts `ENV COMPOSER_NO_DEV=1` as a valid alternative.

**Violation:**

```dockerfile
FROM php:8.4-cli AS app
WORKDIR /app
COPY composer.json composer.lock ./
RUN composer install --no-interaction
```

**Fix:**

```dockerfile
FROM php:8.4-cli AS app
WORKDIR /app
COPY composer.json composer.lock ./
RUN composer install --no-dev --no-interaction
```

**Equivalent (using env var):**

```dockerfile
FROM php:8.4-cli AS app
ENV COMPOSER_NO_DEV=1
RUN composer install
```

**Configuration:**

```toml
[rules.tally."php/composer-no-dev-in-production"]
severity = "warning"
```
